\begin{abstract}
We present a systematic empirical evaluation of \emph{PyNear}, a Python
library implementing exact and approximate $k$-nearest-neighbour
($k$-NN) search via Vantage-Point Trees (VP-Trees) and an IVF-style
approximate index with SIMD-accelerated distance kernels.  Version 2.1 introduces
(i) a BLAS-backed inner-cluster flat scan for \texttt{IVFFlatL2Index}
using the NumPy SGEMV identity
$\|q-x\|^2 = \|q\|^2 + \|x\|^2 - 2\,q^\top X$,
replacing the previous per-cluster VP-Tree traversal,
(ii) a native C++ Lloyd's K-Means implementation with K-Means++ initialisation,
SIMD L2 distances, and OpenMP parallelism, which replaces the previous
scikit-learn dependency for clustering and yields a
$2.4\text{--}3.7\times$ build-time speedup, and
(iii) multiple exact-index query optimisations: best-first (min-heap) tree
traversal, cache-friendly data reordering, FMA distance kernels, improved
vantage-point selection by spread maximisation, and \texttt{int32\_t} node
indices for reduced memory pressure — collectively yielding a $3.3\times$
query-latency reduction at $d=512$.
PyNear is benchmarked against three widely adopted libraries —
\emph{Faiss}, \emph{scikit-learn}, and \emph{Annoy} — across five
experimental configurations spanning the Euclidean ($\ell_2$),
Manhattan ($\ell_1$), and Hamming distance metrics, with dataset
cardinalities up to $N = 10^6$ and dimensionalities from
$d = 2$ to $d = 1024$.  In the exact-search regime,
\texttt{VPTreeL2Index} achieves query latencies up to $220\times$
lower than scikit-learn's kd-tree and up to $39\times$ lower than Faiss
\texttt{IndexFlatL2} at high dimensionality ($d=512$);
at low dimensionality ($d \le 16$) it is $2\text{--}4\times$
faster than Faiss's BLAS-backed scan.  In the high-dimensionality approximate regime
($d = 128\text{--}1024$), the BLAS-optimised \texttt{IVFFlatL2Index}
achieves competitive latency (5.7–19.2 ms for 32 queries at $n_{\mathrm{probe}}=20$),
outperforming Faiss \texttt{IndexIVFFlat} at $d=128$ and $d=512$.
Build time is now \emph{faster} than Faiss IVF for $N \le 50\,000$
and within $1.4\text{--}5\times$ at larger scales owing to the C++ K-Means kernel.
All experiments were conducted on an
Intel\textsuperscript{\textregistered}
Core\textsuperscript{\texttrademark} Ultra 9 285K (24 cores).
\end{abstract}

\section{Introduction}
\label{sec:intro}

Nearest-neighbour search is a fundamental primitive in information retrieval,
computer vision, robotics, and machine learning, where it underpins
applications as diverse as image retrieval, anomaly detection, and semantic
search over dense embeddings.  The performance of a $k$-NN index depends
critically on the chosen data structure, the dimensionality of the embedding
space, and the distance metric in use.

PyNear addresses these challenges through a C\texttt{++} core built around
\emph{Vantage-Point Trees}~\cite{yianilos1993data}, a metric-space
tree that avoids the curse-of-dimensionality limitations of classical
kd-trees by exploiting the triangle inequality rather than axis-aligned
partitioning.  SIMD intrinsics (AVX2 on x86-64) further accelerate the
hot distance computation paths.  The library exposes a uniform Python API for
both exact \texttt{VPTree*} indices and the approximate \texttt{IVFFlat*}
family, which implements an inverted file (IVF) scheme over Voronoi cells
with a configurable probe budget.

Version 2.1 replaces the per-cluster VP-Tree traversal inside
\texttt{IVFFlat*} with a vectorised flat scan using NumPy's BLAS-backed
matrix--vector product (SGEMV).  For $\ell_2$, the identity
$\|q - x\|^2 = \|q\|^2 + \|x\|^2 - 2\,q^\top X$ reduces inner-cluster
distance computation to a single SGEMV call, with per-cluster squared norms
pre-computed at build time.  This approach provides a $2\text{--}4\times$
speed-up within each probed cluster compared to tree traversal at high
dimensionality.

The remainder of this report is organised as follows.
Section~\ref{sec:setup} describes the experimental setup.
Section~\ref{sec:exact} reports results for exact-search indices across
$\ell_2$, $\ell_1$, and Hamming metrics.  Section~\ref{sec:approx}
evaluates the approximate regime.  Section~\ref{sec:build-scale}
analyses build-time scalability from $N=10\,000$ to $N=10^6$.
Section~\ref{sec:discussion} synthesises the findings, and
Section~\ref{sec:conclusion} concludes.

\section{Experimental Setup}
\label{sec:setup}

\paragraph{Hardware.}
All experiments were executed on a workstation equipped with an
Intel\textsuperscript{\textregistered} Core\textsuperscript{\texttrademark}
Ultra 9 285K processor (24 cores, AVX-512 capable).  No GPU acceleration was
used; all libraries ran on the CPU.

\paragraph{Competitors.}
We compare against the following external libraries:
\begin{itemize}
  \item \textbf{Faiss}~\cite{johnson2019billion} (\texttt{FaissIndexFlatL2},
        \texttt{FaissIndexBinaryFlat}, \texttt{FaissIVFL2}): a production-grade
        library from Meta AI, BLAS-accelerated, supporting both exact and
        approximate indices.
  \item \textbf{scikit-learn}~\cite{pedregosa2011scikit} (\texttt{SKLearnL2}):
        a general-purpose machine-learning toolkit whose nearest-neighbour
        implementation is based on ball-trees and kd-trees.
  \item \textbf{Annoy}~\cite{bernhardsson2018annoy} (\texttt{AnnoyL2},
        \texttt{AnnoyManhattan}, \texttt{AnnoyHamming}): an approximate index
        based on random projection trees, optimised for memory-mapped
        read-only access.  Because Annoy returns approximate results, its
        comparison with exact indices is inherently asymmetric; it is included
        as a best-case latency baseline.
\end{itemize}

\paragraph{Protocol.}
Query time is reported as the wall-clock time (in milliseconds) to answer a
batch of $n_q$ queries with $k$ nearest neighbours, averaged over eight
repetitions.  Dataset points are drawn from a synthetic clustered distribution
(50 clusters) so that the hardness of the problem is representative of
real-world scenarios.  The exact parameters for each benchmark are reported
in the caption of the corresponding figure.

\section{Approximate-Search Benchmarks}
\label{sec:approx}

PyNear's \texttt{IVFFlatL2Index} partitions the data into Voronoi cells via
K-Means and at query time probes only $n_{\mathrm{probe}}$ of them, trading
recall for speed.  Version 2.1 replaces the per-cluster VP-Tree traversal
with a flat BLAS scan using the NumPy SGEMV identity and introduces a native
C++ Lloyd's K-Means (K-Means++ initialisation, SIMD L2, OpenMP) for clustering:
\[
  \|q - x\|^2 = \|q\|^2 + \|x\|^2 - 2\,q^\top X,
\]
where $q^\top X$ is a BLAS SGEMV call $(n_c, D) \times (D,) \to (n_c,)$
with per-cluster squared norms $\|x\|^2$ pre-computed at build time.
This section compares \texttt{IVFFlatL2Index} against Faiss's
\texttt{IndexIVFFlat} (canonical IVF-style approximate index) and the Faiss
exact flat baseline. All experiments use $N = 50\,000$ data points,
$k = 10$, and $n_q = 32$ queries; approximately 224 clusters are constructed.

\section{Discussion}
\label{sec:discussion}

\paragraph{Low-to-moderate dimensionality ($d \leq 64$).}
VP-Trees offer the best exact-search latency in this regime across all three
distance metrics tested ($\ell_1$, $\ell_2$, $\ell_\infty$).  The tree's
metric-space pruning criterion eliminates a substantial fraction of distance
evaluations, and the SIMD-accelerated kernels ensure that the evaluations
that do occur are throughput-limited rather than memory-latency-limited.
scikit-learn's kd-tree degrades rapidly beyond $d = 8$ due to increasing
overlap between ball partitions; Faiss's flat scan is memory-bandwidth-bound
and grows steadily with $d$.

\paragraph{High dimensionality ($d = 128\text{--}512$) — exact.}
\texttt{VPTreeL2Index} remains the most efficient exact method at $d=512$
(2.2 ms vs.\ 85.4 ms for Faiss flat, 483.2 ms for scikit-learn), a $39\times$
advantage over Faiss.  The best-first search strategy and cache-friendly data
layout (introduced in v2.1) deliver a $3.3\times$ speedup at $d=512$
compared to the prior DFS-based traversal.  The tree's pruning effectiveness
degrades with dimension — a manifestation of the concentration of measure
phenomenon — but remains significant for clustered datasets even at 512 dimensions.

\paragraph{High dimensionality ($d = 128\text{--}1024$) — approximate.}
The BLAS-optimised \texttt{IVFFlatL2Index} (v2.1) delivers compelling
approximate-search performance.  At $d=128$ it outperforms Faiss
\texttt{IndexIVFFlat} (5.7 ms vs.\ 7.1 ms) and at $d=512$
(9.3 ms vs.\ 7.8 ms) while maintaining perfect Recall@10.
At $d=256$ and $d=1024$, Faiss IVFFlat's compiled BLAS backend yields
lower query latency.  Build time with the C++ K-Means++ kernel is
\emph{faster} than Faiss IVF at $N \le 50\,000$ and within
$1.4\text{--}5\times$ at larger scales.
For workloads where $N \ge 250\,000$ and build time is critical,
Faiss IVF remains the faster option; at smaller $N$, or when a pure-Python
install (NumPy only) or the exact-to-approximate fallback via a single
parameter change are operationally valuable, \texttt{IVFFlatL2Index}
is the preferable choice.

\paragraph{Binary descriptors.}
Faiss's flat binary index is the fastest exact binary search method at
all descriptor lengths.  \texttt{VPTreeBinaryIndex} remains
viable for moderate lengths ($\leq 64$ bits, $< 6$ ms) and for workloads
where tree-based pruning amortises the per-query cost.
\texttt{BKTreeBinaryIndex} is best suited to range queries
(\texttt{find\_threshold}), where its B-K tree structure is optimal, and
should not be interpreted as a general-purpose $k$-NN competitor.

\section{Conclusion}
\label{sec:conclusion}

PyNear's VP-Tree family provides highly competitive exact $k$-nearest-neighbour
search across $\ell_1$, $\ell_2$, and $\ell_\infty$ distance metrics,
delivering the lowest latency among all evaluated exact methods across the
full tested dimensionality range ($d = 2\text{--}512$) without requiring BLAS,
GPU, or any native runtime dependency beyond NumPy.
Version 2.1's exact-index improvements (best-first traversal, cache-friendly
layout, FMA kernels, spread-maximising VP selection) deliver a $3.3\times$
query-latency reduction at $d=512$: from 7.3 ms to 2.2 ms, compared to
85.4 ms for Faiss \texttt{IndexFlatL2} and 483 ms for scikit-learn.  The SIMD-accelerated
distance kernels ensure that all three metric variants operate within a
narrow, near-identical latency band ($\pm 15\%$), allowing practitioners to
select the semantically appropriate metric without performance penalties.

Version 2.1 introduces two major improvements to \texttt{IVFFlatL2Index}:
a BLAS-backed inner-cluster flat scan (NumPy SGEMV), enabling competitive
query performance — outperforming Faiss \texttt{IndexIVFFlat} at $d=128$ and
$d=512$ while maintaining perfect Recall@10 at $n_{\mathrm{probe}} = 20$ —
and a native C++ Lloyd's K-Means (K-Means++ init, SIMD L2, OpenMP) that
replaces the prior sklearn dependency and delivers build times
$1.2\text{--}1.8\times$ faster than Faiss IVF at $N=50\,000$, and
$4.7\times$ faster at $N=10\,000$.  At larger scales ($N \ge 100\,000$)
Faiss's compiled K-Means is faster, with the gap widening to ${\sim}5\times$
at $N=10^6$.

Future work includes extending the SIMD kernels to AVX-512 for wider
vectorisation, investigating product-quantisation-based inner-cluster
compression to reduce the latency of high-dimensional IVFFlatL2Index probes, and
evaluating performance on non-clustered (adversarial) data distributions.

\paragraph{Static index limitation.}
A notable architectural constraint of the current implementation is that all
PyNear indices are \emph{static}: the full tree must be reconstructed via
\texttt{set(data)} whenever the underlying dataset is modified.  No
incremental insertion, deletion, or in-place point update is supported.
This precludes direct use in workloads where the dataset evolves continuously,
most notably real-time physics simulation.  In particle-based simulation
systems (SPH fluids, cloth, soft-body dynamics), nearest-neighbour queries
must reflect the current particle positions after every integration step;
rebuilding a VP-Tree at each frame is prohibitively expensive.  Production
physics engines address this with structures designed for dynamic updates:
dynamic Bounding Volume Hierarchies (DBVH), incremental kd-trees, or spatial
hashing with temporal coherence.  Dynamic VP-Tree variants that support
$O(\log n)$ insertions and deletions have been studied in the
literature~\cite{yianilos1993data} and represent a natural direction for
future development of PyNear.
